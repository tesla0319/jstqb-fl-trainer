## 1. システム概要

### 1.1 背景と目的

既存の SQL Silver 学習支援アプリをベースに、JSTQB Foundation Level (FL) 向け学習支援アプリへ改修する。

MVP の目的は「JSTQB FL 向け問題を出題し、回答し、正誤判定と解説表示ができること」に絞る。  
過剰設計・追加機能の先行実装は行わない。

### 1.2 対象資格

| 項目 | 内容 |
|------|------|
| 対象 | JSTQB Foundation Level (FL) |
| 非対象 | Advanced Level / Test Analyst / Technical Test Analyst / Test Manager / Agile Tester / Automotive Tester |

### 1.3 既存資産の扱い

| 資産 | 扱い |
|------|------|
| SQL Silver 学習支援アプリ | 保守のため既存リポジトリとして残す |
| JSTQB FL 版 | 新規リポジトリとして管理 |
| FastAPI 構成 | 流用する |
| pytest 構成 | 流用する |
| 複数選択ロジック | Phase 1 では削除しない（既存コードへの影響最小化のため） |

> **複数選択ロジックについての補足**  
> JSTQB FL 版では使用しない。seed 問題にも含めない。UI 上でも表示しない。  
> Phase 2 以降で不要が確定した場合に削除を検討する。

### 1.4 システム構成（概略）

```
[ブラウザ]
    ↕ HTTP
[FastAPI (Backend)]
    ↕
[SQLite / DB]
```

- フロントエンド: 既存 UI を流用（HTML / CSS / JS）
- バックエンド: FastAPI (Python)
- データストア: SQLite（既存構成に準拠）
- テスト: pytest

## 2. 機能一覧

### 2.1 MVP で実装する機能

| # | 機能 | 概要 |
|---|------|------|
| F-01 | 問題出題 | JSTQB FL の問題を出題する |
| F-02 | カテゴリ指定出題 | 指定カテゴリから問題を出題する |
| F-03 | ランダム出題 | 全カテゴリからランダムに出題する |
| F-04 | 選択肢表示 | 4択の選択肢を表示する |
| F-05 | 回答送信 | ユーザーが選択肢を選んで送信する |
| F-06 | 正誤判定 | 選択した回答の正解 / 不正解を判定する |
| F-07 | 解説表示 | 回答後に解説を表示する |
| F-08 | カテゴリ一覧取得 | 問題カテゴリの一覧を返す |
| F-09 | 難易度管理 | 問題ごとに難易度を保持する |

### 2.2 MVP で実装しない機能（Phase 2 以降）

| 機能 | 理由 |
|------|------|
| 模擬試験モード（40問・制限時間・合格判定） | 複雑性が高い。MVP スコープ外 |
| 学習履歴・分析 | DB 拡張が必要 |
| AI 問題生成 | 外部サービス連携が必要 |
| ユーザー管理 / 認証 | インフラ要件が増える |
| UI 大幅変更 | MVP では既存流用で十分 |
| DB 構造の大幅変更 | 既存資産への影響リスク |

### 2.3 問題形式の制約

| 項目 | 採用 / 非採用 | 備考 |
|------|-------------|------|
| 単一選択問題 | ✅ 採用 | JSTQB FL 本番試験形式に合わせる |
| 正解は 1 つのみ | ✅ 採用 | 同上 |
| 原則 4 択 | ✅ 採用 | 同上 |
| 複数選択（2つ選べ 等） | ❌ 非採用 | MVP スコープ外 |
| 複数正解 | ❌ 非採用 | 同上 |
| 順不同採点 | ❌ 非採用 | 同上 |

|---------|--------|--------|

### 3.2 画面一覧

| 画面 ID | 画面名 | 概要 |
|---------|--------|------|
| SCR-01 | トップ / 出題設定画面 | カテゴリ選択・出題開始 |

```
┌──────────────────────────────────────┐

│                                      │

│  ○ 選択肢 B                          │

```
┌──────────────────────────────────────┐

- 「次の問題へ」: 同じ設定で次の問題を出題（Phase 1 では同じ問題が再出題される場合がある）
- 「終了」: SCR-01 へ戻る

## 4. データ仕様

### 4.1 カテゴリ定義

Phase 1 で使用するカテゴリは以下の 6 つとする。

| カテゴリID | カテゴリ名 | JSTQB シラバス対応章 |
|-----------|-----------|-------------------|
| 1 | テストの基礎 | 第1章 |
| 2 | テスト活動とプロセス | 第2章 |
| 3 | 静的テスト | 第3章 |
| 4 | テスト技法 | 第4章 |
| 5 | テストマネジメント | 第5章 |
| 6 | ツール支援 | 第6章 |

### 4.2 難易度定義

| 難易度値 | ラベル | 表示 |
|---------|--------|------|
| 1 | 基礎 | ★ |
| 2 | 標準 | ★★ |
| 3 | 難しめ | ★★★ |

### 4.3 問題データ仕様

Phase 1 では 10 問を用意する。

#### 問題データの条件

- オリジナル問題であること（公式問題のコピー禁止）
- JSTQB FL シラバス準拠の内容であること
- 全問にカテゴリ・難易度・解説を設定すること
- 全問が単一選択・4 択であること

#### Phase 1 問題配分

全 6 カテゴリに最低 1 問を含める。

| カテゴリ | 問題数 |
|---------|--------|
| テストの基礎 | 2 |
| テスト活動とプロセス | 2 |
| 静的テスト | 1 |
| テスト技法 | 2 |
| テストマネジメント | 2 |
| ツール支援 | 1 |
| **合計** | **10** |

### 4.4 DBテーブル定義

> **Phase 1 の DB 方針**
>
> - `categories` テーブルは作成しない。カテゴリ一覧は `config.JSTQB_CATEGORIES` から返す。
> - `questions.category` 文字列でカテゴリを管理する既存方式を維持する（正規化は Phase 2 以降）。
> - `choices.display_order` は既存カラム名をそのまま使用する（`order_num` への改名は行わない）。

#### questions テーブル

| カラム名 | 型 | NOT NULL | 説明 |
|---------|----|----------|------|
| id | INTEGER | ✅ | 主キー（AUTO INCREMENT） |
| category | TEXT | ✅ | カテゴリ名文字列（`config.JSTQB_CATEGORIES` の name と一致させる） |
| difficulty | INTEGER | ✅ | 難易度（1 / 2 / 3） |
| question_text | TEXT | ✅ | 問題文 |
| multi_select_count | INTEGER | ✅ | 選択数（Phase 1 では常に 1。既存ロジックとの後方互換のため維持） |
| explanation | TEXT | ✅ | 解説文 |
| trap_reason | TEXT | ❌ | 間違えやすいポイント（NULL 可） |
| created_at | DATETIME | ✅ | 作成日時 |

#### choices テーブル

| カラム名 | 型 | NOT NULL | 説明 |
|---------|----|----------|------|
| id | INTEGER | ✅ | 主キー（AUTO INCREMENT） |
| question_id | INTEGER | ✅ | 問題 ID（FK: questions.id） |
| choice_text | TEXT | ✅ | 選択肢テキスト |
| is_correct | BOOLEAN | ✅ | 正解フラグ（1問につき1つのみ true） |
| display_order | INTEGER | ✅ | 選択肢の表示順（1〜4）。**Phase 1 ではシャッフルせず昇順固定で表示する** |

#### categories テーブル

Phase 1 では **作成しない**。カテゴリ一覧は `app/config.py` の `JSTQB_CATEGORIES` 定数から返す。

```python
# config.py
JSTQB_CATEGORIES = [
    {"id": 1, "name": "テストの基礎"},
    {"id": 2, "name": "テスト活動とプロセス"},
    {"id": 3, "name": "静的テスト"},
    {"id": 4, "name": "テスト技法"},
    {"id": 5, "name": "テストマネジメント"},
    {"id": 6, "name": "ツール支援"},
]
```

> **`choices.is_correct` の制約方針**
>
> Phase 1 では DB 制約（UNIQUE 制約等）は追加しない。理由は以下の通り。
>
> - MVP では過剰設計を避ける
> - 初期問題数が少なく手動確認が容易
> - seed データと pytest で品質を担保できる
> - DB 設計変更による影響範囲を広げたくない
>
> 代わりに pytest で以下を必ず確認すること。
>
> - 全問で `is_correct=true` の選択肢が **必ず 1 件** 存在する
> - `is_correct=true` が **0 件** の問題が存在しない
> - `is_correct=true` が **2 件以上** の問題が存在しない

### 5.3 GET `/questions` — 問題一覧取得

#### クエリパラメータ

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| category_id | integer | ❌ | 指定時: 該当カテゴリのみ返す。未指定時: 全カテゴリ対象 |
| random | boolean | ❌ | `true` のとき **`limit` の値に関係なく** 1問をランダムに返す。デフォルト: false |
| limit | integer | ❌ | 取得件数上限。デフォルト: 10。`random=true` のときは無視される |

#### 0件時の挙動

対象問題が 0 件の場合（`category_id` 絞り込みで該当なし 等）は、**空配列 `[]` を返す**。  
理由: カテゴリ絞り込みで 0 件になることは異常ではなく、クライアント側で「該当する問題がありません」と表示すればよいため。404 より扱いやすい。

```json
{ "questions": [] }
```

#### 出題済み問題の扱い（Phase 1）

Phase 1 では出題済み問題の履歴を保持しない。  
同じカテゴリ・同じ条件で「次の問題へ」を押した場合、**同じ問題が再出題される可能性を許容する**。  
出題履歴管理は Phase 2 以降（学習履歴機能）で対応する。

#### レスポンス

```json
{
  "questions": [
    {
      "id": 1,
      "category_name": "テストの基礎",
      "difficulty": 2,
      "question_text": "ソフトウェアテストの主な目的として最も適切なものはどれか。",
      "choices": [
        { "id": 1, "choice_text": "バグを完全になくすこと", "display_order": 1 },
        { "id": 2, "choice_text": "欠陥を発見すること", "display_order": 2 },
        { "id": 3, "choice_text": "ソフトウェアを動作させること", "display_order": 3 },
        { "id": 4, "choice_text": "開発を高速化すること", "display_order": 4 }
      ]
    }
  ]
}
```

> **Phase 1 実装メモ**: レスポンスフィールドは `category_name`（`questions.category` 文字列から生成）と `display_order`（`choices.display_order` カラムをそのまま使用）。`category_id` はレスポンスには含まない。

> **注意**: `choices` には `is_correct` を含めない。  
> 理由: 出題時点でクライアントに正解を渡さないため。ブラウザの通信内容（DevTools 等）から正解が見える状態を避ける。  
> 正解情報は `/answers` レスポンスにのみ含める。

### 5.5 POST `/answers` — 回答送信・正誤判定

#### リクエストボディ

```json
{
  "question_id": 1,
  "choice_id": 2
}
```

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| question_id | integer | ✅ | 回答する問題の ID |
| choice_id | integer | ✅ | 選択した選択肢の ID |

#### レスポンス

```json
{
  "is_correct": true,
  "correct_choice_id": 2,
  "correct_choice_text": "欠陥を発見すること",
  "selected_choice_id": 2,
  "selected_choice_text": "欠陥を発見すること",
  "explanation": "テストの主な目的は欠陥を発見することであり、..."
}
```

| フィールド | 型 | 説明 |
|-----------|-----|------|
| is_correct | boolean | 正解なら true |
| correct_choice_id | integer | 正解の選択肢 ID |
| correct_choice_text | string | 正解の選択肢テキスト |
| selected_choice_id | integer | ユーザーが選んだ選択肢 ID |
| selected_choice_text | string | ユーザーが選んだ選択肢テキスト |
| explanation | string | 解説テキスト |

#### サーバー側バリデーション

回答送信時に、指定された `choice_id` が指定 `question_id` に属しているかを検証する。  
属していない場合は `400 Bad Request` を返す。

理由: 回答ミスではなく不正なリクエストであるため、サーバー側で整合性を保証すべき。

#### エラーレスポンス

| ステータス | 条件 |
|-----------|------|
| 400 | `choice_id` が指定 `question_id` に属していない |
| 404 | `question_id` または `choice_id` が存在しない |
| 422 | リクエストボディの型が不正 |

400 エラー時のレスポンス例:

```json
{
  "detail": "choice_id does not belong to the specified question_id"
}
```

## 6. 非機能要件

### 6.1 動作環境

| 項目 | 内容 |
|------|------|
| Python バージョン | 既存リポジトリに準拠 |
| フレームワーク | FastAPI（既存流用） |
| DB | SQLite（既存流用） |
| テストフレームワーク | pytest（既存流用） |
| 実行環境 | ローカル開発環境（Phase 1）|

### 6.2 テスト方針

pytest を継続利用する。Phase 1 で確認するテスト観点は以下の通り。

#### 必須テスト観点

| テスト観点 | 確認内容 |
|-----------|---------|
| カテゴリ取得 | 全 6 カテゴリが正しく返ること |
| 問題取得（全件） | 問題一覧が返ること |
| 問題取得（カテゴリ指定） | 指定カテゴリの問題のみ返ること |
| 問題取得（ランダム） | 1 問が返ること |
| 問題取得（0 件） | 空配列 `[]` が返ること |
| 問題取得（存在しない ID） | 404 が返ること |
| 回答送信（正解） | `is_correct: true` と解説が返ること |
| 回答送信（不正解） | `is_correct: false` と解説が返ること |
| 回答送信（question_id / choice_id 不存在） | 404 が返ること |
| 回答送信（choice_id が別問題に属する） | 400 が返ること |
| 回答送信（型不正） | 422 が返ること |

#### seed データ品質確認テスト

| テスト観点 | 確認内容 |
|-----------|---------|
| is_correct=true が 1 件のみ | 全問で正解選択肢が必ず 1 件だけ存在すること |
| is_correct=true が 0 件の問題なし | 正解のない問題が存在しないこと |
| is_correct=true が 2 件以上の問題なし | 正解が複数ある問題が存在しないこと |
| 全カテゴリに問題が存在する | 6 カテゴリすべてに最低 1 問含まれること |

#### 補足（境界値・例外ケース）

- `limit=0` や負値を渡した場合の挙動（Phase 1 では未定義。実装時に決定する）

### 6.3 設定値（config 化候補）

以下の値はハードコードせず、設定ファイルや環境変数での管理を推奨する。

| 設定項目 | Phase 1 の値 | 備考 |
|---------|------------|------|
| 問題取得の `limit` デフォルト値 | 10 | 将来的な変更に備えてハードコードを避ける |
| 難易度の最大値 | 3 | 段階追加の可能性があるため |
| DB ファイルパス | `app.db`（既存構成に準拠） | Phase 1 では変更しない。環境分離が必要になった場合に config 化を検討する |
